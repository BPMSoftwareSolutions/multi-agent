Verification status on this checkout:

- Passing: `test:evidence`, `test:taxonomy-swarm-report`, `test:codebase-story-review-report`, `test:readme-projection`, `test:story-coherence-package`, `test:ascii-components`.
- Failing: `test:taxonomy-case`, `test:taxonomy-heal`, `test:taxonomy-heal-run`, `test:taxonomy-scan-report`.

The failures are useful evidence. They show that several fixtures and report paths still encode the old false model: "method responsibility differs from file responsibility, therefore incoherent." Under the honest metric, distinct method responsibilities are good; copied, placeholder, duplicate, or merged responsibilities are bad.

## Governance Principles

1. Do not hand-stitch production responsibility text across the studio codebase.
2. Hand edits are acceptable for evaluator logic, test fixtures, and deterministic plumbing.
3. Production responsibility content should be generated by a governed worker operation, validated by the deterministic evaluator, and recorded in a ledger.
4. Coherence must reward specific, distinct, single responsibilities. It must penalize copied file responsibilities, duplicate sibling responsibilities, placeholder text, bare identifiers, and merged responsibilities.
5. Missing taxonomy and present-but-false taxonomy must remain separate governance states.

## Key Findings From Code Review

### 1. The evaluator flip is only partial

`src/story-analysis/coherence-evaluator.js` uses the new integrity-based scoring, but `src/story-analysis/coherence-aggregator.js` still imports `./coherence-analyzer`, which still uses the old similarity metric. Any codepath through the aggregator can still produce the false narrative score.

Next action:

- Route all coherence analysis through one evaluator.
- Either delete/retire `coherence-analyzer.js` or make it a compatibility wrapper around `coherence-evaluator.js`.
- Update formatters that still expect `issue.similarity`, `issue.threshold`, and `issue.flags`.

### 2. Issue schema consumers still expect similarity fields

The new evaluator emits `issue.reasons`; several surfaces still render old fields:

- `cli/taxonomy-case-file.js` formats `similarity` and `threshold`.
- `cli/taxonomy-heal-run.js` summarizes `similarity`.
- `src/story/narrative-builder.js`, `src/story-analysis/analysis-formatter.js`, and `src/story-analysis/report-formatter.js` still assume similarity/flags.

Next action:

- Define a stable issue schema for honest coherence:
  - `method`
  - `methodResp`
  - `reasons`
  - `isBoilerplate`
- Add a shared formatter such as `formatCoherenceIssue(issue)` so report code does not duplicate assumptions.

### 3. Expected remediation still generates copied method responsibilities

`cli/taxonomy-case-file.js` currently builds `expectedMethods` by assigning the same synthesized responsibility to every method. Under the new evaluator, that remediation can never honestly earn 100 because it creates duplicate sibling responsibilities and often copies the file responsibility.

Next action:

- Change expected remediation so the file responsibility is a concise aggregate claim.
- Give each method a distinct responsibility derived from the method name and detected behavior.
- Keep target score 100 only when the expected taxonomy truly satisfies the evaluator.

### 4. Scan reporting conflates score-zero false taxonomy with missing taxonomy

`src/observability/taxonomy-scan-report.js` currently treats `!file_anchor_found || file.score === 0` as `missing_taxonomy`. A copied-responsibility file can have anchors and score 0; that should be classified as weak or false taxonomy, not missing taxonomy.

Next action:

- Classify missing only when the file anchor is absent.
- Classify score 0 with present anchors as weak/false taxonomy.
- Keep finding and next-action language distinct:
  - missing taxonomy: add anchors
  - false taxonomy: split or repair responsibilities

### 5. The old "maximize overlap" packet must be retired

`coherence-surgical.json` still instructs the model to "maximize vocabulary overlap with file anchor vocabulary." That is the old bad incentive. Running it now would manufacture the exact anti-pattern the new evaluator is meant to catch.

Next action:

- Do not run `coherence-surgical.json` against the studio.
- Replace it with a `split_responsibility` packet that asks for distinct, specific method responsibilities.

## Phase 1: Stabilize Honest Coherence

Goal: make the current evaluator change coherent across the whole repo and get the verification suite green.

Tasks:

1. Consolidate evaluator entry points.
   - Make `coherence-aggregator.js` use `coherence-evaluator.js`.
   - Decide whether `coherence-analyzer.js` becomes a wrapper or is removed from active paths.

2. Normalize evaluator behavior.
   - Compare copied and duplicate responsibilities case-insensitively after whitespace normalization.
   - Keep placeholder detection deterministic.
   - Treat zero-method files as explicitly "no methods to validate."
   - Consider making merged-responsibility detection more direct than only `/ and [A-Z]/`.

3. Update issue rendering.
   - Replace old similarity text with reason-based text.
   - Example: `readProgress: copied_from_file, duplicate_of_sibling`.

4. Fix the four failing tests with honest fixtures.

   - `test:taxonomy-heal`: same fixture principle.
   - `test:taxonomy-heal-run`: same fixture principle plus status story text updated for reason-based issues.
   - `test:taxonomy-scan-report`: present score-zero taxonomy should not be counted as missing taxonomy.

5. Fix expected remediation generation.
   - Do not generate the same method responsibility for every method.
   - Generate distinct expected method responsibilities for tests.
   - Keep production mass wording for the bee, not manual edits.

## Phase 2: Build The `split_responsibility` Worker Operation

Goal: create a governed operation that rewrites copied, duplicate, placeholder, identifier-like, and merged method responsibilities into distinct method responsibilities.

Recommended files:

- `src/worker-bee/operations/split-responsibility.js`
- `cli/worker-bee-split-responsibility.js`
- `packets/split-responsibility.studio.packet.json`
- `tests/verify-split-responsibility.js`

Work item shape:

```json
{
  "path": "src/example.js",
  "file_responsibility": "Coordinates example behavior",
  "current_score": 40,
  "issues": [
    {
      "method": "readProgress",
      "responsibility": "Coordinates example behavior",
      "reasons": ["copied_from_file"]
    }
  ],
  "methods": [
    {
      "name": "readProgress",
      "line": 42,
      "code_excerpt": "function readProgress() { ... }"
    }
  ]
}
```

The model should only supply semantic wording. The operation should deterministically:

- discover impacted files from the evaluator,
- package file and method evidence,
- request distinct responsibility text,
- validate the proposed output before writing,
- apply only anchor blocks,
- preserve shebangs, line endings, and unrelated source bytes,
- re-run extraction and coherence after each write,
- reject any proposal that lowers score, duplicates siblings, copies the file responsibility, introduces placeholders, or creates merged responsibilities.

Suggested model output shape:

```json
{
  "files": [
    {
      "path": "src/example.js",
      "methods": [
        {
          "name": "readProgress",
          "responsibility": "Parses worker progress logs into completed packet counts"
        }
      ]
    }
  ]
}
```

Important guardrails:

- The file responsibility should usually remain unchanged during `split_responsibility`.
- Method responsibilities must be plain descriptive clauses, not symbol names.
- Method responsibilities must be distinct from the file responsibility and from each other.
- The operation should run dry by default unless `--apply` is present.
- If a file cannot reach 100 by anchor rewriting alone, emit a `refactor_required` ledger row instead of forcing text.

Acceptance gate:

```powershell
npm run test:split-responsibility
node cli/worker-bee-split-responsibility.js --target tests --dry-run --json
node cli/worker-bee-split-responsibility.js --target tests --apply --json
npm run test:evidence
npm run test:taxonomy-case
npm run test:taxonomy-heal
npm run test:taxonomy-heal-run
```

## Phase 3: Run A Controlled Studio Cleanup

Goal: point the bee at the studio codebase and repair the copied/merged responsibility backlog without random hand edits.

Steps:

1. Baseline.
   - Run `npm run taxonomy-coherence-scan -- .` or the equivalent scanner path.
   - Store the baseline report under `reports/`.
   - Count files below 100 and classify reasons.

2. Pilot.
   - Run `split_responsibility` on 5 to 10 files.
   - Use `--dry-run` first.
   - Apply only if every proposed file improves or stays stable and no unrelated source body changes.

3. Batch.
   - Run against the studio root in small batches.
   - Exclude `.git`, `node_modules`, `reports`, `.tmp`, `coverage`, and generated artifacts unless explicitly included.
   - Write one ledger per run.

4. Regenerate governed artifacts.
   - Re-run taxonomy scan reports.
   - Re-run codebase story review.
   - Regenerate any fixtures that are intended to reflect bee-produced output.

5. Verify no hand-stitch drift.
   - Inspect `git diff --stat`.
   - Confirm changed production source lines are anchor comments only unless a file is explicitly marked `refactor_required`.
   - Confirm report artifacts match the current scan.

## Phase 4: Commit Strategy

Use small, reviewable commits:

1. `honest coherence evaluator`
   - extractor fix,
   - evaluator consolidation,
   - issue schema and report compatibility,
   - four fixture/test repairs,
   - regenerated governed reports.

2. `split responsibility worker operation`
   - operation implementation,
   - packet,
   - CLI,
   - unit/fixture tests,
   - dry-run/apply verification.

3. `bee generated studio responsibility cleanup`
   - generated anchor text changes,
   - split-responsibility ledger,
   - regenerated reports.

Do not include the hand-edited `cli/fractal-taxonomy-scanner.js` demonstration in the mass cleanup commit unless it is regenerated by the bee or explicitly retained as part of the first stabilization commit.

## Definition Of Done

The next stage is done when:

- The evaluator has one active scoring definition.
- The full verification suite passes.
- Tests encode real incoherence, not the old "distinct means bad" assumption.
- Missing taxonomy and false taxonomy are separate report states.
- `coherence-surgical.json` is retired or replaced.
- `split_responsibility` exists as a governed, dry-run-first operation.
- A pilot run proves anchor-only changes can raise coherence without source-body mutation.
- The studio cleanup is backed by a ledger and regenerated evidence artifacts.
